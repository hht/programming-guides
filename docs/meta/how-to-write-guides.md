# 如何编写世界级开发指南（Guide of Guides）

> **本文是元指南**：规定「以后如何产出某一品类的开发指南」。 
> **工程方法契约**：[../../AGENTS.md](../../AGENTS.md)（极简 / SSOT / TDD / 根因优先）。 
> **领域指南**（如 [../defi/](../defi/README.md)）才是给实现 agent 用的规格包。 
> 写领域指南时：**不要**往指南仓堆业务实现；**不要**默认依赖自有产品仓当唯一标杆。

## 1. 目标与成功标准

### 1.1 目标

产出一份领域指南，使得实现 agent 在仅持有：

- 该领域指南 
- 设计稿 
- API / 后端契约 
- （若适用）链上合约地址与 ABI / 其它领域特有契约 

时，能在**新仓库从零自写**出达到或超越该类**世界级开源标杆**的工程与功能面。

### 1.2 「世界级」覆盖面（必做 vs 参考）

| 维度 | 要求 | 验收证据（写入领域 acceptance） |
|------|------|----------------------------------|
| 目录组织 | **必做** | `02` 有树 + 依赖方向；禁类型大口袋 |
| 命名 / 词表 | **必做** | **业务优先**：领域/实体/业务操作 = 代码与目录名；`02` 含领域词表（或指向 `UBIQUITOUS_LANGUAGE.md`）；协议字段冻结；**语法（snake/kebab）只作第二层**；禁 `*Dto`/`*Manager`/`handle*` 等技术翻译名进领域。强制块：[naming-business-first.md](./naming-business-first.md) |
| 代码风格 | **必做** | 语言：经 [language-gates/<lang>.md](./language-gates/README.md) 落实 §0（MUST/MUST NOT）+ `commands` lint 绿；`01` 只链 gate、不抄全文。框架：`00` 含 **框架 MUST** 表（F01…）；横切册语言跟宿主、不另开闸。显式>隐式；禁无关炫技 |
| 工具链 | **必做** | 脚手架命令 + lockfile 策略 |
| 门禁 | **必做** | `commands.md`：PR check / 发版 e2e |
| 代码极简 | **必做** | 禁令：指南无业务实现；deletion-first |
| 逻辑清晰可测 | **必做** | domain 纯函数；门闸单测 case 表 |
| 算法 / 关键路径 | **必做** | 核心路径步骤规格 + 探针 |
| 测试 | **必做** | case→期望；场景×断言矩阵 |
| 安全（应用层） | **必做** | 品类相关会话/授权/注入等规格 |
| 无障碍 / 性能预算 | **宜做** | 默认可裁；若裁须在 acceptance 写「裁剪：理由」一行；不裁则须有预算数字或检查项 |
| **运维 / 第三方可观测** | **仅参考** | 可写「若接入则…」；**不进必勾** |

**环境成对配置**（staging/prod API+契约）属 **INPUTS/应用配置**，不是「对接 Sentry 类运维第三方」，仍为必做。

### 1.3 「达到 / 超越」操作定义

对选定的证据源集合 \(B\)（默认 3 个；开源为仓，P1w 为公开文档）：

1. **能力切条**：差距表「功能面」每一行 = 一条**用户可感知能力**（例：连接钱包、SIWE 登录、Swap 提交）。禁止把整站当一条。 
2. **共有判定**：能力出现在 \(B\) 中 **≥2** 个证据源的公开能力描述中（开源看 README/docs/主要用户流；P1w 看该 URL 文档）。仅 1 源独有 → 默认「可选」。 
3. **功能面达到**：指南必做功能 ⊇ 所有「共有」能力。 
4. **工程面达到**：§1.2 每个**必做**维有对应章节 + acceptance 勾选。 
5. **超越**（须同时满足 **a+b**；若证据源 \(B\) **全部为 P1w**，另加 **c**）： 
 - **a.** 核心正确性路径列出 ≥2 条更硬不变量；每条格式：`对照：B 中未见/更弱 → 本指南要求 …` 
 - **b.** 发版有场景×断言矩阵 
 - **c.**（仅全 P1w）「更硬」必须相对**已引用的公开段落**可核对，禁止无对照的自我声明 
6. **写入领域 acceptance**：`11-world-class-acceptance.md` 必须**复制**本小节 1–5 为可勾清单（可改措辞，不可删判定条件）；对抗时以领域文件为准、以本元指南为冲突仲裁。

禁止「合理并集 / 中位数」等无法打勾的表述。

## 2. 硬性禁令

1. **指南 ≠ 项目**：禁止提交可运行业务模块。 
2. **缺源码 ≠ 对抗 FAIL**：只审规格是否够自写。 
3. **禁止自有仓当唯一标杆**：外部标杆为主；自有仓仅反例/教训。 
4. **templates/** 仅允许：JSON Schema、OpenAPI stub、env 例、架构 lint 例、状态矩阵 md、**`package-scripts.snippet.json`（仅 scripts 键名与命令字符串，非可运行业务）**、**`templates/README.md`**。 
5. **运维第三方**不进必勾验收。 
6. **禁止臆造**：无 P0/P1/P1w 依据不得写入必做规格。

## 3. 标准流程

```text
定品类 → 选标杆(3) + P0 → 差距表 → 确定默认栈 → 写 docs/<domain>/
 → 对抗 LOOP(5×) → 只修规格至 5/5 PASS → 提交
```

### 3.1 定品类

一句话：用户 / 核心动作 / 做完标准。 
列出 **1 条核心正确性路径**名称（全文唯一主路径）。

### 3.2 选标杆（默认 3 开源）

**入选门槛（须同时满足；缺一则换仓或走降级）：**

- 与品类一句话匹配（同用户任务，不是仅同技术词） 
- 近 12 个月有实质提交，或仍为生态事实标准 
- 可只读学习（许可证允许阅读；指南不粘贴受限代码） 
- 技术栈可映射到候选默认栈（或接受在 sources 写明映射成本） 

```bash
gh search repos "<品类关键词>" --sort stars --limit 15
gh repo clone org/name /tmp/bench-<name> -- --depth 1
```

记入 `sources.md`：

| ID | 仓库或文档 | 等级 | 学什么 | 不学什么 | 品类匹配一句 |
|----|------------|------|--------|----------|--------------|

**开源不足时的降级：**

| 情况 | 允许 |
|------|------|
| 找到 ≥3 开源 | 必须用 3 开源为 P1 标杆 |
| 仅 1–2 开源 | 开源全用上；缺额用 **P1w** 补足至合计 **3** 个证据源 |
| 0 开源 | **3** 个 P1w；见下方「收窄」 |

闭源公开文档 / 工程博客 = **P1w**（不是 P1）。P1 仅开源仓或一线开源协议工程文。

**「收窄必做面」（仅 0 开源或 P1w≥2 时）：**

- **工程必做维**（§1.2）：**不可砍**，仍须全覆盖 
- **功能必做**：仍用 §1.3 的「能力切条 + ≥2 证据源共有」；**额外**禁止把任何证据源都未文档化的能力标成必做（不得臆造） 
- 即：收窄 = 禁止臆造，**不是**把共有门槛降成「≥1 即可」 
- acceptance **所有必做功能项**均须带 `sources` URL（不论是否触发收窄）

**P1w 硬门闸：** 不得臆造；与 P0 冲突以 P0 为准；必做功能项必须能指向具体公开 URL。

另必列 **P0** ≥3 条。`sources.md` 最少：P0≥3、证据源合计=3、冲突记录表（可空）。

### 3.3 差距表（强制列）

文件：`docs/<domain>/sources.md` 或 `research-notes.md`。

| 缺口 | 来自标杆 | 类型(功能/工程) | 落入文件 | 必做/可选/参考 |
|------|----------|-----------------|----------|----------------|
| … | A/B/C | … | `05-….md` | 必做 |

运维第三方 → 类型=参考。

### 3.4 默认栈决策（固定步骤）

**选栈总则：先进优先，兼顾可映射；禁止「下载量第一」单独定胜负。**

1. 列出 3 标杆技术栈交集与可映射项，以及各层的 **P0 候选**（官方当前主推 / 类型安全与工程边界更强者） 
2. 在「可映射到标杆能力面」的前提下，选 **一套** 默认栈（构建+UI+领域库），裁决顺序： 
 1. **工程先进性**（端到端类型、明确 SSOT 边界、与当前 React/语言代际对齐、可测性） 
 2. **P0 官方主推或一线工程共识**（文档、生态、持续维护） 
 3. **流行度**（npm 周下载、采用面）— 仅作佐证；**当 1 与 3 冲突时取 1** 
3. 写入 `01-stack-*.md`：脚手架命令、锁版本策略；写入 `sources.md` 冲突表（例：流行库 vs 更先进库的取舍一句） 
4. 仅当 3 标杆无交集且无可信 P0 时，才降级到「标杆多数派」并在 sources 说明 

**示例（React SPA）：** `react-router-dom` 下载更高，但 **TanStack Router** 在路由/search/loader 端到端类型上更先进 → **采用 TanStack Router**（流行度记入冲突表，不改裁决）。

禁止留下「Vite 或 Next 任选」「Router A 或 B 任选」类开口。

### 3.5 写领域指南

按 §4。核心路径 = **编号步骤规格**（可含类型签名级伪代码），禁止完整业务文件。

**「够自写」深度标准（每章自检）：**

- 实现 agent 读完后，对关键决策 **无需再问「选 A 还是 B」** 
- 有门闸/失败分类/键名/默认数值处必须写明或指向 INPUTS 必填项 
- 禁止「产品定」而不给默认；若需产品定，写入 INPUTS 必填勾选 

### 3.6 对抗 LOOP（可直接开跑）

> **操作 SSOT**：[adversarial-review.md](./adversarial-review.md)（触发、缺角、真伪、prompt、收工）。 
> 下文为摘要；冲突以 `adversarial-review.md` 为准。

#### 配置

| 项 | 值 |
|----|-----|
| 数量 | **5**（一人一角，禁止合并） |
| 模型 | 默认 **`grok-4.5-fast-xhigh`**；降级规则见对抗方案 |
| 上下文 | 每次**新开**干净 Task；禁止 resume 审同一轮 |
| 通过 | **5/5 PASS**；缺角未齐不得裁 DONE |
| FAIL 处理 | **只改规格**；禁止往指南加业务实现消 FAIL |
| 轮次 | 无硬上限；同一真 BLOCKER 连续 2 轮未消 → 停手升级人工 |

#### 宿主开跑步骤

1. 确认 §7 清单 1–6 已勾（领域指南草稿齐套）；重审则跳过新品类项但须确认变更已落盘 
2. 按 [adversarial-review.md](./adversarial-review.md) §4：**并行 5 Task**（缺角补跑规则见该文 §4.3） 
3. Prompt = 对抗方案 §6 骨架 
4. 收齐 5 份 `VERDICT` → 主裁（格式见对抗方案 §4.4） 
5. 未 5/5：改规格 → **新 ROUND** 全新 5 路 
6. 5/5 后勾 §7 第 8 项（对抗）并提交 

#### 五角（固定）

| # | 视角 |
|---|------|
| A | Agent 执行协议 / 闭环 |
| B | 核心正确性路径规格 |
| C | INPUTS / 契约 / 地址或等价 SSOT |
| D | 目录·**业务命名/词表**·UI 状态交付 |
| E | 测试矩阵 / commands / acceptance |

完整 prompt、真伪 BLOCKER 表、Focused 预检变体 → [adversarial-review.md](./adversarial-review.md)。

### 3.7 提交

```text
docs(<domain>): <一句话 why>
```

更新 [../README.md](../README.md)。分支随意；勿提交 `.DS_Store`。不强制 PR 模板。

## 4. 领域指南固定产出

路径：`docs/<domain>/`

**编号规则：** `00` 原则 → `01` 栈 → `02` 目录 → **`03`–`08`** 核心域（最多 6 个文件；不够则合并主题，禁止占用 09–11）→ 固定尾号：`09-testing-ci.md`、`10-checklist.md`、`11-world-class-acceptance.md`。`INPUTS.md` / `commands.md` / `sources.md` / `README.md` 不编号。

| 文件 | 必含 |
|------|------|
| `README.md` | 执行协议步骤、默认栈一句、索引表 |
| `INPUTS.md` | 设计/API/环境/品类契约；缺则停 |
| `00-principles.md` | 品类 + 核心路径 + **框架 MUST**（MUST/MUST NOT + 探针）+ SSOT 表 |
| `01-stack-*.md` | 脚手架、锁版本、Compiler/等价 |
| `02-directory-and-naming.md` | 目录树、依赖方向、**业务命名词表（Pass 1）**、语法表（Pass 2） |
| `03+` | 每文件：步骤规格或契约；差距表行有落入 |
| `09-testing-ci.md` | 单测 case→期望；e2e 场景×断言；CI 何时 |
| `10-checklist.md` | 开工顺序勾选 |
| `11-world-class-acceptance.md` | 内嵌 §1.2 必做维勾选表 + §1.3 共有/超越勾选；运维第三方不进必勾 |
| `commands.md` | 命令与 exit 0 |
| `sources.md` | P0/P1/P1w/标杆表 |
| `templates/README.md` | 允许文件清单 |
| `templates/*` | 仅 §2 允许类 |

**不依赖 defi 实例即可齐套。** 可选对照 [../defi/](../defi/README.md) 的文件布局。 
**禁止**在固定清单外新增同级 `*.md`（研究笔记进 `sources.md` 附录；临时草稿不入库）。

#### `03+` 单文件最小骨架（复制填空）

```markdown
# NN — <主题>

## 不变量
- …

## 步骤规格（实现自写）
1. …
2. …

## 失败分类 / 默认值（须写明，或指向 INPUTS）
| 情况 | 行为 |

## 单测探针（case → 期望）
| case | 期望 |
```

差距表示例行：

| 缺口 | 来自 | 类型 | 落入文件 | 必做/可选/参考 |
|------|------|------|----------|----------------|
| 提交前模拟交易 | A,B | 功能 | `05-….md` | 必做 |

### 4.1 INPUTS 最小门闸

设计 frame+状态；API+错误码；环境成对；品类契约；**默认授权/等价策略勾选**（若适用）。

### 4.2 测试最小集

核心门闸单测表；发版场景×断言；PR `check` exit 0。指南**不附**可运行测试源码。

## 5. 证据等级

| 等级 | 含义 |
|------|------|
| **P0** | 官方规范 / 框架 / EIP 等 |
| **P1** | 世界级**开源**仓实践、开源协议工程文 |
| **P1w** | 闭源产品公开文档 / 博客（证据弱） |
| **E** | 本元流程约定 |

冲突：P0 > P1 > P1w > E。

## 6. DeFi 实践教训（摘要）

| 固定 | 禁止 |
|------|------|
| INPUTS + 执行协议 | 指南堆业务实现 |
| 核心路径步骤规格 | 自有仓当唯一标杆 |
| 对抗=够自写；FAIL 只修规格 | 用加代码消 FAIL |
| P0 对照 | 运维第三方当必达 |
