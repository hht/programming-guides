# AGENTS.md — 世界级软件工程契约（通用）

> **给谁用**：在本仓或任意目标仓工作的 AI 编码助手与人类协作者。  
> **是什么**：跨语言、跨产品的工程决策与行为契约；**不是**某一业务产品的实现说明。  
> **领域指南**：具体栈/品类规格见 [`docs/`](./docs/README.md)；如何写指南见 [`docs/meta/how-to-write-guides.md`](./docs/meta/how-to-write-guides.md)。  
> **冲突裁决**：本文件（通用方法）> 领域指南（栈规格）> 临时聊天约定。产品仓可另有更细的 `AGENTS.md`，但不得削弱本节不变量。

---

## 1. 第一性原理

1. **先问对不对，再问快不快**：每一步只问——该不该做、当前是否错、是否服务用户可观察目标。  
2. **正确架构不应产此类 bug**：出问题先问结构 / SSOT / 边界为何允许；优先消除结构条件（常是减法），而不是加特判。  
3. **根因 > 补丁**：禁止用 if/guard/吞异常/硬编码旁路掩盖症状。快改止血须显式标记 `症状补丁-deferred` 并记下结构债与后续路径。  
4. **可证伪才算完成**：没有验收标准、失败模式或验证路径的工作，不算做完。  
5. **暴露不确定性**：涉及数据所有权、同步、schema/API、安全边界、性能目标时，禁止静默假设；先写清假设与缺口。

**决策优先级（钉死）**：

```text
正确性 > 可验证性 > 简洁性 > 复用 > 速度
```

仅当已证明不可行才停；「难 / 大 / 贵 / 边缘」不是停手理由。

## 2. 世界级工程不变量

### 2.1 代码极简（Simplicity / deletion-first）

- 优先最直接、最容易验证的数据流；清晰优于精巧；显式优于隐式。  
- **项目内一致性**优先于外部「理想模式」。  
- 抽象必须立刻降低认知负担，或保护真实边界；否则不引入。  
- 禁止无必要的 wrapper / adapter / `*Manager` / `*Service` 套壳。  
- **deletion-first**：做完后删除、内联或收窄不服务不变量、成功标准或验证路径的状态、字段、类型、分支、配置与 helper。  
- 小而精准：不做机会主义重写、无关重命名、无关格式化。

### 2.2 SSOT（单一真相源）

下列每类真相只能有一个 owner：

| 真相类 | 要求 |
|--------|------|
| 业务规则 / 不变量 | 一处定义；调用方不复制规则 |
| 配置语义 | 一处解析；禁平行第二套 env 含义 |
| 状态 / 会话 | 一处写入权威；派生值标明 Derived |
| Schema / 协议字段 | 一处契约；实现与文档同构 |
| 命名 / 词表 | 业务词 = 代码名；冻结对外字段 |

发现重复逻辑：先回收到现有 SSOT，不要默认再包一层。

### 2.3 DRY / KISS / YAGNI

- **DRY**：只消除真实且稳定的重复；不为去重制造脆弱抽象。  
- **KISS**：能读懂、能测、能删，优于「更通用」。  
- **YAGNI**：没有当前不变量、成功标准或验证路径的未来能力不做。

### 2.4 逻辑清晰、可测试

- 纯领域规则进可单测的函数/模块；I/O、框架、UI 适配放边缘。  
- 门闸与核心路径有 **case → 期望** 表；发版有场景 × 断言矩阵。  
- 失败可分类（用户错 / 校验错 / 依赖错 / 内部错），禁止「假成功」。

### 2.5 算法与核心路径精妙

- 每个系统有一条（或极少几条）**核心正确性路径**（钱、权限、一致性、发布、Agent turn…）。  
- 路径写成编号步骤规格 + 失败分类 + 探针；禁止只靠口头「注意一下」。  
- 「精妙」= 边界正确、复杂度受控、可证明，不是炫技或过早优化。

### 2.6 TDD / 测试纪律

默认路径（行为变更时）：

```text
复现或失败测试（红）→ 最小实现（绿）→ 重构（保持绿）
```

- Bug/回归：未复现或无失败测试前，不写「修复」（快改止血除外，且须 deferred）。  
- 测试描述行为与边界，不锁实现细节。  
- 指南仓不堆可运行业务套件；目标仓须有可执行门禁（`check` / 等价）。

### 2.7 安全与配置（应用层最低线）

- 密钥不入库；环境成对（staging/prod 结构同构、值分离）。  
- 授权/会话边界显式；禁止用「看起来像登录」代替真实会话。  
- 供应链：依赖可锁定；CI 引用宜钉版本或 commit SHA。

### 2.8 无障碍与性能（宜做）

- 有 UI 则默认可达性门闸（对比度、焦点、标签等按领域指南）。  
- 性能：先预算与探针，再优化；禁止无测量的「感觉优化」。

## 3. 工作方式（对 AI 与协作者）

### 3.1 默认端到端

理解问题 → 读相关上下文 → 最小充分修改 → 验证 → 报告。  
用户明确要求「只分析 / 暂不修改 / 只给建议」时，停在分析，不写盘。

### 3.2 先定义再实现

写代码前钉清：真实问题、约束、边界、失败模式、验收标准。  
切片绑定一个用户可判断的 case，或一个可证明的边界。

### 3.3 报告必须含

改了什么、为什么、如何验证（pass/fail 或未跑原因）、剩余风险。  
常规及以上任务：报告末附一段**反面意见**（何处仍可能错）。

### 3.4 提交与 git

无用户明确要求 → 不提交、不推送、不 amend、不改 git 配置、不做破坏性历史改写。  
不覆盖用户未提交改动。

### 3.5 影响面（共享模块）

改跨模块共享逻辑前：评估调用方与风险；CRITICAL 停手；HIGH 须确认后写盘；LOW/MEDIUM 报告后可写。  
工具名与流程以目标仓文档为准；无图谱时人工列 call site，禁止混用两套 impact 工具装懂。

## 4. 根因分类（诊断时必填一句）

修 bug 前用一句话归类（可多选主类）：

| 类 | 含义 |
|----|------|
| SSOT 分裂 | 两处真相不一致 |
| 边界泄漏 | 层职责不清，状态乱写 |
| 契约漂移 | schema/API/事件与实现不符 |
| 时序/并发 | 竞态、假成功、重复提交 |
| 配置/环境 | 环境不成对或默认危险 |
| 工具/认知缺口 | 缺探针、缺文档、静默假设 |
| 纯疏忽 | 单点笔误（仍须防再发） |

禁止在未归类时声称「根因已明」。

## 6. 收工清单（缺一项 = 未完成）

**轻量**（纯文案/单文件无行为）：①改了什么 ②一句验证。  

**常规**（默认）：轻量 + ③为什么 ④验证 pass/fail ⑤剩余风险 ⑥反面意见。  

**高影响**（协议/大重构/安全边界等）：常规 + ⑦根因分类 ⑧是否仍会复发 ⑨若做了多 agent 对抗：仲裁与探针结论。

---

## 7. 一句话纪律

> 用第一性原理找根因，用 SSOT 收真相，用测试钉行为，用减法保极简；补丁只可暂缓，不可当作完成。
