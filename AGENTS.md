# AGENTS.md

> 给 AI 助手的行为契约。处理任务前先读本文件。  
> 栈/品类规格 → [`docs/`](./docs/README.md)；写指南 → [`docs/meta/how-to-write-guides.md`](./docs/meta/how-to-write-guides.md)。

## 决策序

```text
正确性 > 可验证性 > 简洁性 > 复用 > 速度
```

## 工作方式

- **端到端**：理解 → 改 → 验证 → 报告。用户说「只分析」则不写盘。
- **先定义再实现**：问题、约束、失败模式、验收标准先钉清。
- **最小闭环**：每刀绑定一个用户可判断的结果或边界。
- **根因 > 补丁**：禁症状特判；快改须标 `症状补丁-deferred` + 结构债。
- **小而精准**：禁无关重写/重命名/格式化。
- **暴露不确定**：数据所有权、同步、契约、安全、性能 — 禁止静默假设。
- **报告**：改了什么 / 为什么 / 如何验证 / 剩余风险。

## 工程不变量

- **SSOT**：规则、配置语义、状态、契约字段、词表 — 各只有一个 owner。
- **DRY**：只去稳定重复；不为去重造脆弱抽象。
- **KISS / YAGNI / deletion-first**：能删则删；无验收的未来能力不做。
- **清晰 > 精巧；显式 > 隐式**；项目内一致 > 外部理想模式。
- **可测**：领域规则可单测；核心路径有 case→期望；禁假成功。
- **TDD（行为变更）**：红 → 绿 → 重构。未复现不写「修复」（快改除外）。
- **密钥不入库**；环境成对。

## 命名（业务词 = 代码名）

1. 先读目标仓 `UBIQUITOUS_LANGUAGE.md`（或等价词表）；无词表且新业务域 → **先补词表再写代码**。
2. **用业务称呼**：用户说的「订单 / 会话 / 兑换」就是类型、目录、函数名的词根 — 不是 `OrderDto`、`AuthManager`、`DataProcessor`。
3. **禁技术翻译名**进领域：`*Dto` `*Entity` `*Manager` `*Service` `*Helper` `handle*` `process*`（词表未收录则禁）。
4. **禁同义词分叉**：词表是 `Order` 就不要再引入 `Purchase` / `Transaction` 当同一概念。
5. 语法（camelCase / 文件夹 kebab…）只服务已钉死的业务词；基础设施名（`HttpClient`）不得泄漏成 `features/` 主概念。

## 收工

| 档 | 必含 |
|----|------|
| 轻量 | 改了什么 + 一句验证 |
| 常规 | + 为什么 + pass/fail + 风险 + 一句反面意见 |
| 高影响 | + 根因分类 + 是否复发 +（若对抗）仲裁结论 |

无用户明示 → 不 git commit / push。

## 一句话

> 业务词当名字，SSOT 收真相，测试钉行为，减法保极简；补丁只可暂缓。
